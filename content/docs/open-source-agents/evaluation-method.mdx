---

这个页面用于统一开源智能体项目的评估口径。后续每个项目拆解都应按同一套字段写，方便横向比较。

## 评估字段

| 字段 | 要看什么 |
| --- | --- |
| 项目定位 | 它解决代码、研究、浏览器、办公、运维还是通用任务。 |
| 核心流程 | 从用户输入到最终结果，中间经过哪些状态和工具。 |
| 工具系统 | 工具 schema、权限、错误处理、重试、人工确认。 |
| 记忆与状态 | 会话、任务、长期记忆、checkpoint、artifact 如何保存。 |
| 运行环境 | 本地 CLI、Web UI、Docker、远程 sandbox、浏览器或 VNC。 |
| 可观测性 | 是否有 trace、日志、回放、成本统计、失败样例。 |
| 评测与测试 | 是否有测试集、回归流程、benchmark 或人工评审机制。 |
| 安全边界 | 文件系统、Shell、浏览器、网络、密钥、账号如何隔离。 |
| 二次开发 | 代码结构是否清楚，扩展工具、模型和 UI 的成本。 |
| 维护状态 | 最近 release、commit、issue、PR、文档更新和社区活跃度。 |
| 许可证 | 是否允许学习、修改、商用、二次分发。 |

## 拆解模板

每个项目页建议固定包含：

- 项目一句话定位。
- 适合学习的工程点。
- 不适合直接照搬的限制。
- 核心架构图或流程图。
- 关键目录和模块说明。
- 最小运行路径。
- 工具、记忆、状态、UI、评测、安全边界拆解。
- 可借鉴设计和后续验证问题。

## 维护要求

- 项目事实需要写明核验日期。
- 重大版本变化后需要重新核验。
- 如果项目停止维护、许可证不清、运行路径不可复现，应标记为暂缓或淘汰。
