Plandex 是面向大型项目和真实开发任务的开源代码智能体。它强调多步骤计划、长上下文、项目索引和跨文件修改，适合作为“复杂代码任务如何拆解和执行”的样本。

## 一句话定位

Plandex 适合学习“复杂任务计划 + 大上下文项目理解 + 跨文件代码修改”的终端式工作流，尤其适合和 aider、SWE-agent 做对比。

## 值得学习的工程点

- 大任务导向：不是只做小补全，而是处理多步骤、多文件开发任务。
- 项目地图：通过索引和代码结构帮助 Agent 理解大仓库。
- 计划驱动：先形成计划，再逐步修改和审查。
- 终端体验：适合观察 CLI 如何承载复杂 Agent 工作流。
- Go 实现：也适合语言板块里 Go 作为 AI infra/CLI 工具语言的例子。

## 不适合直接照搬的地方

- 长上下文并不自动等于正确，仍需要测试、diff review 和人工确认。
- 大项目索引的成本、准确性和更新策略需要单独评估。
- 终端工具产品化到团队协作场景时，还需要权限、审计和任务状态。

## 后续深拆问题

- Plandex 如何构建和更新项目上下文。
- 它如何把任务拆成计划和执行步骤。
- 它如何处理多文件 diff 和用户 review。
- 它和 aider 的交互模型有什么差异。

## 核心链路

```mermaid
flowchart LR
  U[复杂开发目标] --> P[计划]
  P --> C[项目上下文]
  C --> E[执行修改]
  E --> D[多文件 diff]
  D --> R[用户 Review]
  R --> P
```

Plandex 适合观察“长任务代码修改”怎样维持计划和上下文，而不是每轮都从零开始读仓库。

## 拆解清单

- 计划如何被创建、展示、修改和确认。
- 项目上下文如何索引、更新和裁剪。
- 多文件修改如何保持可 review。
- 用户是否能暂停、回退或要求换方向。
- 终端工作流如何和团队代码评审衔接。
