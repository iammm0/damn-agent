## 这本书解决什么

- 说明代码 Agent 产品的多层架构，而不只停留在 prompt 技巧。
- 拆解查询引擎如何承载 Agent Loop、流式事件和 tool call。
- 分析统一工具接口、专用文件工具和 Bash 安全边界。
- 介绍 Hook、MCP、子 Agent、上下文优化和性能设计。

## 阅读顺序建议

1. 先读架构总览和查询引擎，建立 Agent Loop 中心视角。
2. 再读工具系统和文件工具，理解为什么不用 Bash 包打天下。
3. 然后读子 Agent、Hook、MCP，看扩展边界。
4. 最后读上下文与性能，把「省 token」当成架构能力。

阅读前建议先有 [《Claude Code Harness Engineering》](/docs/resources/harness-engineering-book) 的 Session/Harness/Sandbox 概念；本书更贴近具体产品实现。

- 本书对应 Claude Code v1.8 视角，具体实现可能随版本演进。

## 产品拆解路径

读这本书时建议按“用户看到什么”到“系统如何执行”的顺序：

1. 终端 UI 和命令层：理解产品入口和交互节奏。
2. 查询引擎：理解 Agent Loop、流式事件和 stop reason。
3. 工具系统：理解 schema、权限、Hook 和结果格式化。
4. Bash 与文件工具：理解为什么高风险能力要厚边界。
5. MCP、Hook、子 Agent：理解扩展点和多 Agent 的代价。
6. 上下文与性能：理解 token 预算、缓存和延迟加载。

这个顺序能帮助读者把“源码结构”翻译成“工程设计取舍”。

## 迁移到其他产品

| Claude Code 设计 | 可迁移到 |
| --- | --- |
| 查询引擎 | 任意需要工具调用循环的 Agent 产品。 |
| 统一工具接口 | 浏览器 Agent、研究 Agent、个人助理、代码 Agent。 |
| Bash 安全分层 | 所有能执行命令、调用生产 API 或改文件的系统。 |
| Hook | 团队自定义审计、格式化、测试、权限策略。 |
| MCP | 外部工具、私有数据源、企业系统集成。 |
| 上下文优化 | 长任务、低延迟 UI、高成本模型调用。 |

迁移时不要复制表面 API，要复制边界意识：什么可读、什么可写、什么需要确认、什么必须记录。

## 阅读检查题

- 查询引擎为什么是 Agent Loop 的中心。
- 工具统一接口中哪些字段和安全、并发、缓存有关。
- 为什么 Read/Edit/Write 比 Bash 直接改文件更适合 Agent。
- Hook 和权限规则分别适合解决什么问题。
- 子 Agent 为什么需要 Coordinator，而不是自由聊天。
- 上下文压缩和提示缓存为什么是架构能力。
