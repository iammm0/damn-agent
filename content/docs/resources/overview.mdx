## 维护原则

- 记录资料适合哪个学习阶段，而不是只堆链接。
- 标注是否需要付费、登录或特定运行环境。
- 对高风险领域保留更新时间和核验记录。
- 避免只收藏标题党文章；优先可复核、可反查页码的资料。
- 每本书单独维护介绍页，新增书目时同步更新本页索引表。

## 引用规则

本站以结构化转述为主，不做大段原文搬运：

1. 短引句紧跟「来源：书名，p. 页码」。
2. 表格和段落中的工程结论标注页码范围。
3. 后续扩写优先增加页码出处，而不是增加长引用。
4. 读者应通过书籍页下载 PDF，再按页码回到原文核对。

## 后续扩展

资源清单可按学习阶段过滤：入门、工程化、框架、评测、部署、案例。新增书籍时，在 `meta.json` 注册独立子页，并回本页更新书目表。

## 推荐阅读路线

| 阶段 | 先读什么 | 目标 |
| --- | --- | --- |
| 概念入门 | 《智能体设计模式》 | 建立工具、记忆、规划、多 Agent、MCP、Guardrails 的模式语言。 |
| 工程落地 | 《Claude Code Harness Engineering》 | 理解 Session、Harness、Sandbox、压缩、记忆、权限和恢复。 |
| 产品拆解 | 《Demystifying Claude Code v1.8》 | 观察真实终端编码 Agent 如何组织查询引擎、工具和扩展。 |
| 横向对照 | 框架、开源项目、官方文档 | 把书中模式和真实项目架构对应起来。 |

不要试图一次读完所有资料。更好的方式是：每读一个概念，就回到站内对应页面和一个真实项目页，检查它在工程里如何实现。

## 引用模板

本站后续扩写时，建议使用统一格式：

```md
这个设计把工具调用拆成“模型生成意图、编排层执行、观察结果回填”三步。
来源：《智能体设计模式》，pp. 49-50。
```

如果是转述多页结论，用页码范围；如果是短引句，用单页或具体小节。不要只写“参考某书”，因为读者无法复核语境。

## 资料更新流程

新增资源时按下面流程维护：

1. 保存原始文件或官方链接。
2. 记录页数、版本、作者或来源组织。
3. 新增资源介绍页，并同步 `meta.json`。
4. 在本页书目表增加索引。
5. 在相关概念页或案例页补引用。
6. 如果资料随时间变化，记录核验日期和版本号。

## 质量标准

- 资源应服务学习和工程落地，不收集纯营销内容。
- 优先保留可下载、可引用、可复核页码的资料。
- 官方文档和仓库链接要定期核验。
- 引用外部内容时只做必要短引句，主体使用结构化转述。
- 对高风险主题，例如安全、权限、生产部署，优先使用官方文档和可审计资料。
