Claude Code 是 Anthropic 推出的终端编码 Agent（Coding Agent）。它把 Agent Loop 嵌进开发者日常终端工作流，让模型通过专用工具读改文件、执行命令、搜索代码库，并支持子 Agent、Hook 和 MCP 扩展。

## 一句话定位

Claude Code 适合作为「终端优先的编码 Agent 产品」样本：它不是单纯的聊天补全，而是围绕查询引擎、统一工具接口和权限系统组织出来的工程产品。

## 产品形态

- **终端 UI**：基于 React + Ink 的 CLI 交互，支持流式输出和斜杠命令。
- **Agent Loop**：用户输入触发多轮「推理 → 工具调用 → 观察结果」循环。
- **专用文件工具**：Read、Write、Edit、Grep 等，与 Bash 分工明确。
- **子 Agent**：主 Agent 可派发隔离上下文的子任务执行者。
- **扩展**：Hook 注入业务逻辑，MCP 接入外部工具和服务。
- **项目记忆**：通过 `CLAUDE.md` 等约定文件注入仓库级上下文。

## 值得关注的工程点

| 主题 | 为什么重要 |
| --- | --- |
| 查询引擎 | 承载消息处理、流式事件、tool call 和 stop reason |
| 统一工具链路 | 查找、校验、Hook、权限、执行、格式化结果 |
| Bash 安全 | 高风险命令检测、沙箱、超时和输出截断 |
| 上下文预算 | 压缩、缓存、延迟加载和工具结果截断 |
| Harness 架构 | Session、无状态编排、隔离执行的分工 |

这些主题在站内已有较完整的梳理：[Harness 工程构件](/docs/practices/harness-engineering)、[Claude Code 源码解析](/docs/cases/claude-code-source-analysis)。

## 优势与边界

**适合：**

- 在真实本地仓库里做跨文件修改和命令验证。
- 学习编码 Agent 的工具设计、权限和扩展机制。
- 需要 MCP、Hook 和子 Agent 等较完整扩展面的团队。

**注意：**

- 终端 Agent 对权限、审计和误操作防护要求高，不能只看生成质量。
- 产品能力随版本迭代快，应以官方文档和当前行为为准。
- 深拆实现细节需结合源码观察或逆向资料，本站案例页提供工程化转述。

## 工具与权限边界

Claude Code 的学习价值在于终端 Agent 如何把工具能力做成受控动作，而不是让模型自由输出 shell 脚本：

| 边界 | 要观察什么 |
| --- | --- |
| 文件工具 | Read、Edit、Write 与 Bash 的分工，是否能精确展示 diff。 |
| Bash | 命令确认、超时、输出截断、危险命令识别。 |
| MCP | 外部工具和数据源如何被接入、授权和展示。 |
| Hooks | 团队如何在工具调用前后插入校验、格式化或审计。 |
| 项目记忆 | `CLAUDE.md` 等规则文件如何影响仓库级行为。 |

如果只关注模型回答质量，会漏掉 Claude Code 真正重要的部分：工具调用被放进了可审计、可扩展、可中断的终端工作流。

## 典型任务流

```mermaid
flowchart LR
  U[用户目标] --> S[搜索 / 读取仓库]
  S --> P[计划]
  P --> E[编辑文件]
  E --> T[运行测试或命令]
  T --> R[解释结果]
  R --> D[交付 diff / commit 建议]
```

每一步都应该留下可检查证据：读了哪些文件、改了哪些 hunks、运行了什么命令、失败如何处理、何时需要用户确认。

## 使用边界

- 适合本地仓库探索、跨文件修改、测试驱动修复、重构和文档维护。
- 不适合无审计地执行生产命令、处理真实密钥或自动发布高风险变更。
- 团队使用时应把项目规则、命令白名单、格式化和测试要求写进仓库约定。
- MCP 和 Hook 增强能力的同时也扩大攻击面，需要按最小权限配置。
