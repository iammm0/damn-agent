---

Agent 评测不能只看最终回答是否像样。你需要同时评估路径、工具使用、成本和失败恢复。

单次 demo 很容易成功，因为任务、上下文和预期都由开发者控制。真正上线后，用户会给出模糊目标、过期材料、冲突要求和异常工具返回。评测与回归的作用，是在模型、prompt、工具、检索和权限策略变化时，看清系统质量有没有变好。

## 评测对象

Agent 评测至少分四层：

| 层 | 评测对象 | 例子 |
| --- | --- | --- |
| 输出层 | 最终答案或产物 | 文档是否完整、代码是否通过测试 |
| 过程层 | 计划、工具调用、重试和停止 | 是否先搜索再编辑、是否无限循环 |
| 系统层 | 成本、延迟、稳定性和恢复 | p95 轮数、token、失败后能否回放 |
| 风险层 | 权限、敏感数据和人工接管 | 是否越权写文件、是否泄露密钥 |

不同任务的权重不同。研究型 Agent 更重视引用和证据，Coding Agent 更重视测试和 diff，运维 Agent 更重视权限和回滚。

## 评测维度

| 维度 | 问题 |
| --- | --- |
| 正确性 | 结论是否事实正确？ |
| 完整性 | 是否覆盖关键约束？ |
| 工具使用 | 是否调用了正确工具？ |
| 可复现性 | 失败后是否能回放？ |
| 成本 | token、时间、外部 API 成本是否可接受？ |
| 安全性 | 是否越权或执行高危动作？ |

## 样例集设计

一个好的评测样例不只是输入和标准答案。它应该包含任务背景、固定数据、期望信号、禁止动作和评分方式。

```ts title="agent-eval.ts" lineNumbers
export type AgentEvalCase = {
  id: string;
  task: string;
  fixtures: Record<string, unknown>;
  expectedSignals: string[];
  forbiddenActions: string[];
  maxTurns?: number;
  maxCostUsd?: number;
};

export const cases: AgentEvalCase[] = [
  {
    id: "research-framework-compare",
    task: "比较 LangGraph 和 Mastra 的适用场景",
    fixtures: {},
    expectedSignals: ["状态机", "TypeScript", "人工接管"],
    forbiddenActions: ["编造 benchmark"],
    maxTurns: 8,
    maxCostUsd: 0.5,
  },
];
```

如果样例依赖外部网页或 API，最好把关键材料保存为 fixture。否则每次评测结果会被外部状态影响，无法判断是 Agent 变差还是环境变了。

## 黄金轨迹

对多步骤任务来说，只有最终答案不够。可以保存一条“黄金轨迹”作为参考：

| 轨迹部分 | 检查点 |
| --- | --- |
| 计划 | 是否拆成可执行步骤 |
| 检索 | 是否查了必要资料 |
| 工具 | 是否调用正确工具，参数是否合理 |
| 验证 | 是否运行测试、lint、build 或人工检查 |
| 收尾 | 是否总结变更、风险和未完成事项 |

黄金轨迹不是要求 Agent 每次走完全相同路径，而是定义关键行为信号。例如 Coding Agent 不一定要用同一个搜索词，但必须读取项目约定、只改目标文件、运行匹配测试。

## 回归流程

1. 固定模型版本、工具版本和输入数据。
2. 运行核心评测集。
3. 保存 trace 和摘要指标。
4. 对失败样例做归因。
5. 只在通过阈值后发布。

推荐把回归分成三档：

| 档位 | 触发时机 | 内容 |
| --- | --- | --- |
| 快速集 | 每次 prompt、工具 schema 或代码改动 | 少量关键样例，几分钟内完成 |
| 核心集 | 合并主分支或发布前 | 覆盖主要任务类型和风险路径 |
| 扩展集 | 模型升级、框架升级、权限策略变化 | 长任务、边界样例、历史失败样例 |

## 自动评分与人工评审

自动评分适合检查格式、关键词、结构、测试结果、工具调用和成本阈值。人工评审适合判断事实质量、任务完成度、表达清晰度和复杂场景的取舍。

常见自动评分信号：

- JSON 或 schema 是否通过。
- 是否调用了必要工具。
- 是否避免了禁止工具。
- 测试、lint、build 是否通过。
- 输出是否包含来源、路径或证据。
- token、轮数、耗时是否超出预算。

不要让同一个模型既生成答案又给自己打满分。模型评审可以作为弱信号，但生产闸门应该优先依赖可验证规则、测试和人工抽检。

## 人工评审

自动评测适合做回归闸门，人工评审适合发现新型错误。尤其是研究型、法律、医疗、金融、运维类 Agent，不应该只依赖模型自评。

人工评审最好使用明确 rubric：

| 分数 | 含义 |
| --- | --- |
| 0 | 未完成任务，或出现严重安全问题 |
| 1 | 部分相关，但关键约束缺失 |
| 2 | 基本完成，存在可修复问题 |
| 3 | 完成任务，过程和证据可靠 |

评审意见要能回流成两类资产：新的评测样例，或新的系统约束。只把意见留在聊天记录里，下一次回归不会受益。

## 失败归因

失败样例要分类，否则很难改：

| 类型 | 例子 | 修复方向 |
| --- | --- | --- |
| 任务理解 | 忽略用户边界、误解目标 | 改任务模板和澄清问题 |
| 上下文 | 缺文件、检索错、摘要丢关键约束 | 改上下文选择和压缩 |
| 工具 | 参数错、工具描述重叠、错误语义不清 | 改 schema、描述和错误 envelope |
| 模型 | 规划差、格式漂移、长链路失稳 | 路由到更强模型或拆任务 |
| 权限 | 越权、误拦截、缺审批 | 改权限策略和人工接管 |
| 验证 | 没跑测试、错误测试、只信模型自评 | 强制验证步骤 |

## 发布闸门

发布前可以设置这些门槛：

- 核心样例通过率不低于上个版本。
- 高风险样例不得出现越权动作。
- 工具调用格式通过率达到阈值。
- p95 轮数、p95 成本和失败重试次数不恶化。
- 历史事故样例必须保持通过。
- 新增工具必须有至少一个成功样例和一个失败样例。

## 和可观测性的关系

可观测性负责记录真实轨迹，评测负责重复运行和比较变化。线上失败应该进入评测集，评测失败应该能打开 trace 复盘。两者断开时，团队会不断修同一种问题。

## 最小落地清单

1. 先收集 10 个真实任务，写成固定样例。
2. 给每个样例写成功信号、禁止动作和预算上限。
3. 每次改 prompt、工具 schema 或模型路由都跑快速集。
4. 保存每次评测的 trace、模型版本和工具版本。
5. 把线上失败和人工评审意见沉淀为新样例。
