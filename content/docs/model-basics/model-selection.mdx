---
title: 模型选型与能力边界
description: 按任务类型、上下文长度、延迟、成本、工具能力和多模态需求选择模型。
---

模型选型不是“哪个分数最高选哪个”，而是为你的 Agent 工作流匹配能力层级、成本曲线和失败可恢复性。

同一个模型在聊天、总结、代码修改、工具调用、长上下文检索和多模态任务里的表现可能完全不同。对 Agent 工程来说，选型的关键不是模型品牌，而是任务链路：输入是什么、要调用哪些工具、错误代价多高、能否重试、是否需要人接管。

## 先问五个问题

1. 任务是单次回答，还是多步工具闭环。
2. 需要多长上下文，工具结果会不会反复回灌。
3. 错误代价有多高，能不能接受草稿式输出。
4. 延迟和成本预算如何，是交互式还是批处理。
5. 是否需要视觉、音频、代码专项或结构化输出增强。

这五个问题决定了模型层级。不要先决定模型再硬套场景。

## 能力维度对照

| 维度 | 低成本模型可胜任 | 通常需要更强模型 |
| --- | --- | --- |
| 指令跟随 | 摘要、分类、模板填充 | 复杂约束、多层规则同时满足 |
| 推理深度 | 简单步骤拆解 | 长链路规划、跨文件代码修改 |
| 工具调用 | 单工具、参数简单 | 多工具编排、脆弱参数、恢复重试 |
| 代码能力 | 补全、解释、小范围改动 | 跨仓库重构、测试驱动修复 |
| 上下文 | 短对话、小文档 | 大仓库、长轨迹、多轮失败恢复 |
| 稳定性 | 原型验证、弱约束任务 | 生产回归、低温度下一致行为 |
| 多模态 | 简单图片理解 | 复杂视觉定位、图表解释、屏幕操作 |

“更强模型”不一定意味着所有任务都更好。强模型做简单分类可能只是更贵、更慢；弱模型做高风险多步任务则会放大失败成本。

## 按任务类型选

| 任务 | 模型要求 | 关键验证 |
| --- | --- | --- |
| 分类、标签、路由 | 稳定输出、低成本、低延迟 | schema 通过率、混淆矩阵 |
| 摘要、改写 | 信息保真、长度控制 | 事实遗漏率、引用保留 |
| 研究问答 | 检索整合、引用意识、抗幻觉 | 来源覆盖、结论可复核 |
| 工具调用 | 参数生成、错误恢复、停止条件 | 工具成功率、循环率 |
| Coding Agent | 代码理解、跨文件推理、测试修复 | 测试通过、diff 质量 |
| 浏览器 / Computer Use | 视觉理解、状态跟踪、动作规划 | 任务成功率、误点击率 |
| 高风险业务动作 | 稳定性、保守性、可解释 | 审批触发率、越权率 |

## 常见部署形态

| 形态 | 优点 | 注意点 |
| --- | --- | --- |
| 托管 API | 上手快、版本新、免运维 | 记录 model id、区域、配额和日志策略 |
| 私有化 / 专有云 | 数据边界可控 | 版本滞后、推理性能和工具能力要对齐评测 |
| 开源权重自托管 | 成本可控、可定制 | 需要 GPU 运维、工具调用和结构化能力自行验证 |
| 多模型路由 | 简单任务省钱，难任务升级 | 路由规则、回退链路和评测必须配套 |

## 推荐的分层策略

1. **路由层**：用便宜模型做分类、检索改写、摘要和意图识别。
2. **执行层**：用强模型做规划、代码修改、复杂工具编排。
3. **评审层**：用独立模型或规则做格式检查、风险审查（可选）。
4. **降级层**：强模型失败时，切到更简单路径或人工接管。

不要假设“一个模型包打天下”。Claude Code 等产品也会在查询引擎里处理模型版本、token 预算、stop reason 和流式事件，说明模型只是链路中的一环。来源：《Demystifying Claude Code v1.8》，pp. 62-69。

## 路由示例

```ts title="model-routing.ts" lineNumbers
type TaskKind = "classify" | "summarize" | "tool_agent" | "coding_agent";

type ModelTier = "fast" | "balanced" | "strong";

export function chooseModelTier(task: {
  kind: TaskKind;
  risk: "low" | "medium" | "high";
  estimatedContextTokens: number;
  needsTools: boolean;
}): ModelTier {
  if (task.risk === "high") return "strong";
  if (task.kind === "coding_agent" || task.kind === "tool_agent") return "strong";
  if (task.estimatedContextTokens > 32000) return "balanced";
  if (task.needsTools) return "balanced";

  return "fast";
}
```

这只是示意。真实系统需要用评测数据校准阈值，而不是凭感觉写死规则。

## 版本与回归

模型名不变但权重、系统策略或服务端行为变化时，Agent 行为可能漂移。生产系统应：

- 在 trace 中保存 provider、model、snapshot 或 version 字段。
- 对关键任务维护固定评测集，模型升级先跑回归。
- 对工具调用成功率和格式通过率设阈值，异常时告警或自动降级。
- 记录 prompt 版本、工具 schema 版本和路由规则版本。

模型升级不应该直接等同于质量升级。对 Agent 来说，模型变化可能带来更好的推理，也可能带来新的格式偏差、工具选择偏好和成本曲线。

## 成本和延迟

模型成本不只看单价，还要看任务需要多少轮、每轮上下文多大、工具结果是否反复回灌。

| 决策 | 可能收益 | 风险 |
| --- | --- | --- |
| 强模型一次完成 | 轮数少，复杂任务成功率高 | 单次成本高 |
| 弱模型多轮尝试 | 单轮便宜 | 循环、返工和人工接管成本上升 |
| 路由到多模型 | 成本可控 | 路由错误会影响质量 |
| 长上下文硬塞 | 实现简单 | token 成本高，注意力稀释 |
| 检索后短上下文 | 成本低，可复核 | 检索质量成为瓶颈 |

评估成本时要看“完成一个任务”的总成本，而不是“调用一次模型”的价格。

## 与语言选型的关系

- 产品化 Agent、Web UI、MCP 集成通常优先 [TypeScript](/docs/programming-languages/typescript)。
- RAG、评测、实验性管线通常优先 [Python](/docs/programming-languages/python)。
- 模型本身的能力边界，和语言选型是正交问题：先定任务，再定模型，再定 SDK 和运行时。

## 失败模式

| 决策 | 常见后果 |
| --- | --- |
| 只看榜单分数 | 真实工具链表现不佳 |
| 上下文不够仍硬上 | 频繁压缩、丢约束、成本上升 |
| 强模型做所有子任务 | 成本和延迟失控 |
| 无版本记录 | 线上问题无法复现和回滚 |
| 不跑回归直接升级 | 原本稳定的工具调用开始漂移 |
| 不设降级路径 | 模型失败后只能让用户重试 |

## 选型检查清单

- 任务类型和失败代价是否写清楚。
- 是否有固定评测集，而不是只看 demo。
- 是否记录模型、prompt、工具 schema 和路由规则版本。
- 是否按任务复杂度分层，而不是所有请求走同一模型。
- 是否有成本、延迟、工具成功率和人工接管率指标。
- 高风险任务是否有人工确认或更保守的执行策略。
